# 02 - Interval Greedy

---

# What is Interval Greedy?

Interval Greedy solves problems involving:

```text
Ranges

Intervals

Meetings

Schedules

Time Slots
```

Instead of making decisions on individual elements,

we make decisions on **entire intervals**.

---

# Interval Representation

Most interval problems look like:

```text
[start, end]
```

Example:

```text
[1,3]

[2,6]

[8,10]
```

Think of them as line segments.

```text
1------3

  2--------6

              8------10
```

---

# Core Observation

Almost every interval problem begins with:

```text
Sort the intervals.
```

Why?

Because sorting gives a natural order,

making greedy decisions possible.

# The Three Greedy Ideas

Although there are many interval problems,

they all reduce to one of these ideas.

---

## Pattern 1

Keep the interval that finishes earliest.

Used in:

* LC435 Non-overlapping Intervals
* LC646 Maximum Length Pair Chain

Why?

Finishing earlier leaves more room for future intervals.

---

## Pattern 2

Merge overlapping intervals.

Used in:

* LC56 Merge Intervals
* LC57 Insert Interval

Why?

All overlapping intervals become one larger interval.

---

## Pattern 3

Cover multiple intervals with one action.

Used in:

* LC452 Minimum Number of Arrows
* LC1288 Remove Covered Intervals

Why?

One action can solve multiple overlapping intervals.

---

# Decision Flow

```text

        │
        ▼

        │

────────────────────────────

        │

        │
```

# LC435 - Non-overlapping Intervals

---

# Problem Description

You are given an array of intervals.

Each interval is represented as:

```text
[start, end]
```

Some intervals overlap.

Your task is to **remove the minimum number of intervals** so that the remaining intervals do not overlap.

Return the **minimum number of intervals to remove**.

---

## Example 1

```text
Input

[[1,2],[2,3],[3,4],[1,3]]
```

Output

```text
1
```

Remove:

```text
[1,3]
```

Remaining:

```text
[1,2]

[2,3]

[3,4]
```

No overlaps.

---

## Example 2

```text
Input

[[1,2],[1,2],[1,2]]
```

Output

```text
2
```

Keep only one interval.

# First Thought (Brute Force)

Try removing every interval.

Check whether the remaining intervals overlap.

Repeat for every possibility.

```text
Remove Interval 1

↓

Check

↓

Remove Interval 2

↓

Check
```

Time:

```text
O(2ⁿ)
```

Impossible.

---

# Key Observation

The problem asks:

```text
Minimum intervals to remove
```

Think differently.

Instead ask:

```text
Maximum intervals
that we can KEEP
```

Because

```text
Removed

=

Total

-

Kept
```

This is much easier.

---

# Real Life Intuition

Imagine you are booking meeting rooms.

Meetings:

```text
9-11

10-12

11-1
```

Which meeting should you keep?

The meeting that **finishes earliest**.

Why?

Because it leaves the room free sooner,

allowing more future meetings.

---

# Core Greedy Intuition

Suppose:

```text
[1,5]

[2,3]
```

If you keep:

```text
[1,5]
```

The room is occupied until 5.

Future intervals may not fit.

Instead,

keep:

```text
[2,3]
```

Now the room becomes free much earlier.

This leaves maximum room for future intervals.

---

# Greedy Decision

Always keep the interval with the **earliest ending time**.

Never keep a longer interval if a shorter one ends earlier.

---

# Why Does This Work?

Suppose:

```text
[1,10]

[2,3]

[3,4]

[4,5]
```

Keeping:

```text
[1,10]
```

Only one interval survives.

Keeping:

```text
[2,3]

↓

[3,4]

↓

[4,5]
```

Three intervals survive.

The interval ending earliest always leaves the most opportunities for future intervals.

---

# Algorithm

## Step 1

Sort intervals by:

```text
Ending Time
```

Ascending.

---

## Step 2

Keep the first interval.

Maintain:

```text
previousEnd
```

---

## Step 3

For every next interval:

If:

```text
currentStart >= previousEnd
```

Keep it.

Update:

```text
previousEnd
```

Otherwise,

Remove it.

Increase:

```text
removed++
```

---


# Java Code

```java
class Solution {

    public int eraseOverlapIntervals(int[][] intervals) {

        Arrays.sort(intervals,
                (a, b) -> a[1] - b[1]);

        int removed = 0;

        int previousEnd = intervals[0][1];

        for (int i = 1; i < intervals.length; i++) {

            if (intervals[i][0] >= previousEnd) {

                previousEnd = intervals[i][1];

            } else {

                removed++;
            }
        }

        return removed;
    }
}
```

# Why is this Greedy?

For every overlap,

we immediately choose the interval that ends earlier.

We never reconsider that decision.

That earliest finishing interval leaves maximum space for all future intervals.

This is the classic **Activity Selection Greedy** strategy.
