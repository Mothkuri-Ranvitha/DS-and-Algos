# 01 - Local Greedy Decisions

---

# Variation Definition

The **Local Greedy Decision** pattern solves problems by making the **best possible choice at the current step**, assuming that this choice will lead to the optimal final answer.

Unlike Dynamic Programming, once a Greedy decision is made, **it is never changed**.

---

# Core Intuition

Imagine you're climbing stairs.

At every step, someone asks:

> "What is the best move I can make right now?"

You choose it immediately and never look back.

```text
Current State
      │
      ▼
Best Local Choice
      │
      ▼
Next State
      │
      ▼
Best Local Choice
      │
      ▼
Final Answer
```

Greedy never asks:

> "What if I had chosen something else?"

# Problem 1 — LC455 Assign Cookies

## Problem Description

You have:

* An array `g[]` representing each child's greed factor.
* An array `s[]` representing cookie sizes.

A child is satisfied if:

```text
cookie size >= greed factor
```

Each cookie can be given to only one child.

Find the maximum number of satisfied children.

---

### Example

```text
Greed : [1,2,3]

Cookies : [1,1]
```

Answer:

```text
1
```

Only one child can be satisfied.

---

# First Thought (Brute Force)

For every child,

try every cookie.

```text
Child 1

↓

Check every cookie

↓

Child 2

↓

Check every cookie
```

Time:

```text
O(N²)
```

Too slow.

---

# Key Observation

Suppose:

```text
Children

1 2 3

Cookies

1 2 3
```

Should we give the largest cookie to the smallest child?

No.

That wastes a valuable cookie.

Instead,

give the **smallest cookie that can satisfy the current child**.

This preserves larger cookies for greedier children.

---

# Greedy Decision

Always:

```text
Give the smallest possible cookie
that satisfies the current child.
```

Once assigned,

never change it.

---

# Why Does This Work?

If a child with greed **2** can be satisfied using cookie **2**,

giving cookie **5** instead provides no additional benefit.

Saving larger cookies can satisfy children with higher greed later.

This is the Greedy Choice Property.

---

# Algorithm

### Step 1

Sort both arrays.

```text
Greed

1 2 3

Cookies

1 1 2
```

---

### Step 2

Use two pointers.

* `i` → child
* `j` → cookie

---

### Step 3

If:

```text
cookie >= greed
```

Assign the cookie.

Move both pointers.

Otherwise,

try a larger cookie.

---

# Dry Run

```text
g = [1,2,3]

s = [1,1,2]
```

After sorting:

```text
Children : 1 2 3

Cookies  : 1 1 2
```

Initial:

```text
i = 0

j = 0

answer = 0
```

---

Cookie = 1

Child = 1

Satisfied.

```text
answer = 1

i++

j++
```

---

Cookie = 1

Child = 2

Too small.

```text
j++
```

---

Cookie = 2

Child = 2

Satisfied.

```text
answer = 2

i++

j++
```

Cookies finished.

Final answer:

```text
2
```

---

# Java Code

```java
class Solution {

    public int findContentChildren(int[] g, int[] s) {

        Arrays.sort(g);
        Arrays.sort(s);

        int i = 0;
        int j = 0;

        while (i < g.length && j < s.length) {

            if (s[j] >= g[i]) {

                i++;
            }

            j++;
        }

        return i;
    }
}
```

---

# Complexity

```text
Sorting

O(N log N + M log M)

Scanning

O(N + M)

-------------------

Time

O(N log N)

Space

O(1)
```

---

# First Thought (Brute Force)

For every empty plot,

Try:

```text
Plant

↓

Check Entire Array

↓

Undo

↓

Try Another Position
```

Time becomes very large because we repeatedly check possibilities.

Not needed.

---

# Key Observation

Suppose:

```text
0 0 0
```

Where should we plant?

Middle?

Left?

Right?

It doesn't matter.

Planting at the **earliest valid position** can never reduce future possibilities.

Instead,

it often leaves more room for the remaining plots.

---

# Greedy Decision

Whenever you find a valid position,

```text
Plant Immediately
```

Never postpone the decision.

---

# Why Does This Work?

Suppose:

```text
0 0 0 0
```

If the first position is valid,

waiting to plant later provides no advantage.

Greedy plants immediately and continues.

Once planted,

that decision never needs to change.

---

# Valid Planting Condition

Current plot:

```text
flowerbed[i] == 0
```

Left plot:

```text
i == 0

OR

flowerbed[i-1] == 0
```

Right plot:

```text
i == flowerbed.length-1

OR

flowerbed[i+1] == 0
```

Only then can we plant.

---

# Algorithm

### Step 1

Traverse the flowerbed.

---

### Step 2

For every position,

check:

```text
Current Empty

AND

Left Empty

AND

Right Empty
```

---

### Step 3

If valid,

```text
Plant

flowerbed[i]=1

n--
```

---

### Step 4

If

```text
n==0
```

Return

```text
true
```

---

Otherwise continue.

---



---

```text

```

---

```text

Right = 0
```

Plant.

Flowerbed becomes

```text

```

```text

```

```text

```

---

# Java Code

```java
class Solution {

    public boolean canPlaceFlowers(int[] flowerbed, int n) {

        for (int i = 0; i < flowerbed.length; i++) {

            if (flowerbed[i] == 0) {

                boolean left =
                        (i == 0 || flowerbed[i - 1] == 0);

                boolean right =
                        (i == flowerbed.length - 1 ||
                         flowerbed[i + 1] == 0);

                if (left && right) {

                    flowerbed[i] = 1;

                    n--;

                    if (n == 0)
                        return true;
                }
            }
        }

        return n <= 0;
    }
}
```

---

# Complexity

```text
Time

O(N)

Space

O(1)
```

---

# Why is this Greedy?

Whenever a valid position is found,

we plant immediately.

We never ask:

```text

```

Because planting now never hurts future decisions.

This is exactly the **Local Greedy Decision** property.

# First Thought (Brute Force)

For every customer,

```text

```

Choose one.

---

# Key Observation

Suppose someone pays:

```text

```

```text
$15
```

Two possibilities:

Option 1

```text
10 + 5
```

Option 2

```text
5 + 5 + 5
```

Which one should we choose?

---

# Why Choose 10 + 5?

Because:

```text
A $5 bill is more valuable
than a $10 bill.
```

Why?

A $5 bill is needed to make change for **both**:

* $10 customers
* $20 customers

A $10 bill is only useful for helping make change for a $20 customer.

Therefore,

we should preserve as many **$5 bills** as possible.

---

# Greedy Decision

Whenever possible,

```text
Give

10 + 5

instead of

5 + 5 + 5
```

This keeps more $5 bills for future customers.

---

# Why Does This Work?

Suppose you have:

```text
$5 : 4 bills

$10 : 1 bill
```

A customer pays:

```text
$20
```

If you return:

```text
5 + 5 + 5
```

You lose three $5 bills.

Later,

if someone pays:

```text
$10
```

You cannot return:

```text
$5
```

So using:

```text
10 + 5
```

preserves the smaller denomination.

---

# Algorithm

Maintain only two counts.

```text
five

ten
```

---

For every customer:

### Case 1

Pays:

```text
$5
```

Increase:

```text
five++
```

---

### Case 2

Pays:

```text
$10
```

Need:

```text
$5
```

If no $5 bill exists,

return:

```text
false
```

Otherwise:

```text
five--

ten++
```

---

### Case 3

Pays:

```text
$20
```

Need:

```text
$15
```

First preference:

```text
10 + 5
```

Otherwise:

```text
5 + 5 + 5
```

If neither is possible,

return:

```text
false
```

---

# Java Code

```java
class Solution {

    public boolean lemonadeChange(int[] bills) {

        int five = 0;
        int ten = 0;

        for (int bill : bills) {

            if (bill == 5) {

                five++;

            } else if (bill == 10) {

                if (five == 0)
                    return false;

                five--;
                ten++;

            } else {

                if (ten > 0 && five > 0) {

                    ten--;
                    five--;

                } else if (five >= 3) {

                    five -= 3;

                } else {

                    return false;
                }
            }
        }

        return true;
    }
}
```

We never reconsider previous transactions.

Choosing:

```text

```

instead of

```text
5 + 5 + 5
```

is the local optimal choice because it preserves the most flexible resource ($5 bills).

# LC55 - Jump Game

---

# Problem Description

You are given an array.


# First Thought (Brute Force)

At every index,

try every possible jump.

Example:

```text
Index 0

Jump 1

Jump 2

Jump 3

...
```

Each jump creates more possibilities.

Time becomes:

```text
O(2ⁿ)
```

Impossible.

---

# Better Idea (Dynamic Programming)

Define

```text
dp[i]
```

=

```text
Can I reach the end from index i?
```

Works,

but still requires checking many jumps.

Time:

```text
O(N²)
```

Can we do better?

Yes.

---

# Key Observation

Suppose:

```text
nums =

[2,3,1,1,4]
```

From index 0,

maximum reach is:

```text
0 + 2 = 2
```

You don't actually care which jump you take.

You only care about:

```text
How far can I reach?
```

This completely changes the way we think.

---

# Core Greedy Intuition

Instead of asking:

```text
Which jump should I take?
```

Ask:

```text
What is the farthest index
I can reach so far?
```

Maintain only one variable:

```text
farthest
```

As you move through the array,

keep updating it.

---

# Greedy Decision

At every index:

```text
farthest = max(farthest, i + nums[i])
```

Never commit to a specific jump.

Only remember the **best reach discovered so far**.

---

# Why Does This Work?

Suppose:

```text
nums

2 3 1 1 4
```

Initially:

```text
farthest = 0
```

---

Index 0

Jump:

```text
2
```

Reach:

```text
0 + 2 = 2
```

Now:

```text
farthest = 2
```

---

Index 1

Can we visit it?

Yes,

because:

```text
1 <= farthest
```

Jump:

```text
3
```

Reach:

```text
1 + 3 = 4
```

Update:

```text
farthest = 4
```

We can already reach the last index.

No need to know the exact path.

---

# Failure Condition

Suppose:

```text
nums

3 2 1 0 4
```

Process:

Index 0

```text
farthest = 3
```

---

Index 1

```text
farthest = 3
```

---

Index 2

```text
farthest = 3
```

---

Index 3

```text
farthest = 3
```

---

Index 4

Can we even visit it?

No.

Because:

```text

```

Return:

```text
false
```

---

Maintain:

```text
farthest = 0
```

Traverse every index.

If:

```text

```

```text
false
```

Otherwise,

update:

```text
farthest = max(farthest, i + nums[i])
```

If traversal finishes,

return:

```text
true
```

# Java Code

```java
class Solution {

    public boolean canJump(int[] nums) {

        int farthest = 0;

        for (int i = 0; i < nums.length; i++) {

            if (i > farthest)
                return false;

            farthest = Math.max(farthest, i + nums[i]);
        }

        return true;
    }
}
```
